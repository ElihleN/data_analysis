# Documentation Style Guide

This guide defines the documentation style used in this repository. It is intended to keep project documentation clear, professional, and easy to review.

---

## Documentation Goals

Each project should help a reader understand:

- What problem the project addresses
- What data or inputs are used
- What tools and methods are applied
- How the workflow can be reproduced
- What outputs the project produces
- What skills the project demonstrates

The goal is to make the repository understandable to both technical and non-technical readers.

---

## Recommended README Structure

Each project README should include the following sections:

1. Project title
2. Short project overview
3. Project objectives
4. Business or analytical questions
5. Workflow overview
6. File guide
7. Setup instructions
8. How to run the project
9. Outputs or dashboard summary
10. Skills demonstrated
11. Notes and assumptions
12. Future improvements

---

## Writing Style

Use clear and direct language.

Prefer:

> This script creates analysis-ready tables for monthly, country-level, and customer-level performance.

Avoid:

> This amazing script does some cool SQL things for analysis.

The tone should be professional, practical, and specific.

---

## Technical Documentation Principles

Good technical documentation should be:

- Accurate: describe what the files actually do
- Reproducible: include enough steps for another person to run the work
- Structured: use headings, tables, and short explanations
- Honest: avoid exaggerating results or skills
- Reader-friendly: explain why each step matters

---

## File Documentation Checklist

When documenting a file, include:

- File name
- Purpose of the file
- Required inputs
- Expected outputs
- Dependencies, if relevant
- How it fits into the larger workflow

Example:

| File | Purpose |
|---|---|
| `data_cleaning.sql` | Cleans the raw imported table, converts data types, checks data quality, and creates a cleaned analysis table. |

---

## Project Quality Checklist

Before considering a project portfolio-ready, check that it has:

- A clear README
- Reproducible setup steps
- A logical file order
- No exposed credentials
- Clear explanation of outputs
- Business context or analytical questions
- A future improvements section

---

## Notes for Portfolio Reviewers

This repository is maintained as a learning and professional portfolio. The documentation is intentionally written to show both technical implementation and communication ability.
